## 분석 및 수정 방향 결정

### #1 MemorySaver 싱글턴

**분석**
- 현재는 `chat.py` 모듈 레벨 싱글턴 → `_create_agent()` 가드 → `Agent` 1회 생성 → `MemorySaver` 1회 생성의 3단계 의존성이 맞아야 안전한 구조
- `Agent`가 여러 번 생성되는 상황(테스트, 구조 변경 등)에서는 메모리가 초기화되는 잠재적 버그 존재
- 강사님 권장 방식(모듈 레벨 싱글턴)은 `Agent` 인스턴스 생성 횟수와 무관하게 항상 동일한 메모리를 보장

**수정 방향**: 모듈 레벨 싱글턴으로 변경 ✅

### #2 yfinance news 구조 취약성

**분석**
- yfinance는 Yahoo Finance 비공식 API를 파싱하는 라이브러리로 구조 변경은 피할 수 없는 특성

**수정 방향**: 현상 유지 ✅

### #4 stream_mode 파라미터

**분석**

LangGraph 공식 문서(Streaming Concepts) 확인 결과:
- 공식 지원 stream_mode: `values`, `updates`, `messages`, `custom`, `checkpoints`, `tasks`, `debug` 총 7종
- `checkpoints`, `tasks`, `debug`는 체크포인터 필수 (MemorySaver로 충족)
- 현재 사용 중인 `"updates"`는 ReAct 에이전트의 노드별 상태 변화 추적에 적합한 선택
- 공식 API 시그니처는 `str | list[str]` 지원

검토한 수정 옵션:
- **Option A (채택)**: 파라미터 제거, 내부 `"updates"` 고정 — `agent_service.py`가 항상 `"updates"`를 명시 호출하므로 파라미터 노출 불필요, 캡슐화 관점에서 적합
- **Option B**: `stream_mode: str | list[str] = "updates"` 타입 명시 — 유연성이 필요할 때 적합

**수정 방향**: Option A — `stream_mode` 파라미터 제거, 내부 `"updates"` 고정 ✅

### `app/agents/stock_agent.py` — #1, #4

```python
# #1: 모듈 레벨 싱글턴
_checkpointer = MemorySaver()  # Agent 인스턴스가 여러 번 생성되어도 동일한 메모리를 공유

class Agent:
    def _build_graph(self):
        # ...
        return graph.compile(checkpointer=_checkpointer)  # 싱글턴 재사용

    # #4: stream_mode 파라미터 제거, 내부 고정
    async def astream(self, input_data: dict, config: dict = None):
        async for chunk in self.graph.astream(input_data, config=config, stream_mode="updates"):
            yield chunk
```
