## 학습

### 1. Retriever란?

비구조화 쿼리(자연어)를 받아 관련 문서 목록(`list[Document]`)을 반환하는 인터페이스.

```
입력: str (자연어 질문)
출력: list[Document]
```

- Vector Store와의 차이: Vector Store는 문서 저장+검색이지만, Retriever는 검색만 담당하며 Wikipedia, Kendra 등 외부 소스도 포함하는 더 넓은 개념
- 모든 Vector Store는 `.as_retriever()`로 Retriever로 변환 가능

### 2. ElasticsearchRetriever

Elasticsearch의 Query DSL 전체를 LangChain에서 자유롭게 사용할 수 있게 해주는 범용 래퍼.

**핵심 패턴:**
```python
retriever = ElasticsearchRetriever.from_es_params(
    index_name="인덱스명",
    body_func=쿼리함수,   # str → dict (ES Query DSL 반환)
    content_field="text",
    url="http://localhost:9200",
)
```

`body_func` 하나만 바꾸면 검색 방식을 자유롭게 교체할 수 있음.

### 3. 검색 방식별 요약

| 방식 | 적합한 상황 |
|---|---|
| BM25 (키워드) | 종목명, 날짜처럼 정확한 값 필터링 |
| Vector Search | 의미 기반 유사 문서 검색 |
| Hybrid (BM25 + Vector) | 정확도를 최대로 높이고 싶을 때 |
| Fuzzy Matching | 사용자 입력에 오타가 예상될 때 |
| Complex Filtering | bool 쿼리로 여러 조건 조합 |

주가 OHLCV 데이터는 숫자/날짜 필터 중심이므로 **BM25(term + range 조합)** 이 가장 적합.

### 4. document_mapper

ES 검색 결과(hit)를 LangChain `Document`로 변환하는 방식을 커스터마이징할 수 있음.
LLM이 읽기 좋은 형태로 가공하면 응답 품질이 올라감.

```python
def my_mapper(hit: dict) -> Document:
    src = hit["_source"]
    return Document(
        page_content=f"[{src['ticker']}] {src['date']} | 종가:{src['close']}",
        metadata={"ticker": src["ticker"], "date": src["date"]}
    )
```

## 설계

### yfinance → Elasticsearch 연동 전략 검토

가이드(`docs/ref/elasticsearch_integration_guide.md`)에서 제시된 3가지 트리거링 방식 비교 후 방식 결정.

| 방식 | 설명 | 검토 결과 |
|---|---|---|
| 방식 1: 스케줄링 | 별도 스케줄러가 주기적으로 적재 | 과도한 설정 필요, 학습용에 부적합 |
| 방식 2: On-demand | 요청 시점에 yfinance 호출 후 ES 캐시 | 실시간 주가/뉴스에는 캐시가 오히려 부정확 |
| 방식 3: 하이브리드(TTL) | 마지막 적재 시각 기준 갱신 | 복잡도 높음 |

### 채택 방향: 앱 시작 시 히스토리컬 데이터 적재 + 신규 툴 추가

**결정 근거**
- 실시간 데이터(현재가, 뉴스, 재무정보)는 yfinance 직접 호출이 정확. 기존 3개 툴 유지.
- OHLCV 과거 데이터는 ES에 완벽히 적합. 일별 데이터라 변경이 없어 upsert로 중복 안전.
- 앱 시작 시 1회 적재 → 학습용으로 단순하면서도 ES 핵심 기능(인덱싱, Query DSL, Retriever) 전체 경험 가능.

### 아키텍처

```
[앱 시작]
  → StockDataIngester.ingest_all()
      → yfinance.history("1y") × 4종목 (AAPL, MSFT, TSLA, NVDA)
      → ES bulk upsert (doc_id: {TICKER}_{date})

[사용자 질문]
  → get_stock_history Tool
      → ElasticsearchRetriever (term + range Query DSL)
          → ES index "stock-data"
              → document_mapper → list[Document]
                  → LLM 컨텍스트로 전달
```

### 데이터 규모

| 항목 | 수치 |
|---|---|
| 종목 수 | 4개 (AAPL, MSFT, TSLA, NVDA) |
| 적재 기간 | 1년 (`period="1y"`) |
| 총 문서 수 | 4 × 252 ≈ 1,000건 |
| 예상 적재 시간 | 5~10초 (서버 시작 시) |
| 중복 방지 | doc ID = `{TICKER}_{date}` → upsert |

```python
            "ticker":      {"type": "keyword"},
            "date":        {"type": "date"},
            "open":        {"type": "float"},
            "high":        {"type": "float"},
            "low":         {"type": "float"},
            "close":       {"type": "float"},
            "volume":      {"type": "long"},
            "ingested_at": {"type": "date"}

```

### get_stock_history 툴이 답할 수 있는 질문

| 질문 예시 | Query 전략 |
|---|---|
| "AAPL 최근 30일 종가 알려줘" | term(ticker) + range(date) + sort(date desc) |
| "TSLA 52주 최고가/최저가" | term(ticker) + range(date, 1y) |
| "NVDA 거래량 가장 많았던 날" | term(ticker) + sort(volume desc) + size 1 |

### 검색 전략

주가 OHLCV 데이터는 **숫자/날짜 필터** 중심이므로 BM25 대신 `term` + `range` + `sort` Query DSL 조합을 사용.
임베딩 모델 불필요 → 비용 및 지연 없음.

### document_mapper 설계

```

```

### Query DSL 방식

주가 데이터는 숫자/날짜 필터 중심이므로 BM25 대신 `term + sort` 조합 사용.

```python
{
    "size": days,
    "query": {"bool": {"must": [{"term": {"ticker": ticker}}]}},
    "sort": [{"date": {"order": "desc"}}],
}
```

## 트러블슈팅

### ElasticsearchRetriever 초기화 파라미터 오류

**증상**
```
TypeError: ElasticsearchRetriever.__init__() got an unexpected keyword argument 'es_client'.
Did you mean 'client'?
```

**원인**
가이드 예시 코드에서 `es_client=es_client` 로 작성했으나, `langchain-elasticsearch` 1.x의 실제 파라미터명은 `client`.

**해결**
```python
# 변경 전
ElasticsearchRetriever(..., es_client=es_client)

# 변경 후
ElasticsearchRetriever(..., client=es_client)
```
