## 학습

### Opik Evaluation이란?

- 에이전트의 응답을 정량적으로 측정하는 기능.
- `opik.evaluation.evaluate()`에 데이터셋과 채점 메트릭을 전달하면, 각 항목마다 에이전트를 실행하고 점수를 산출한다.

### 레벨별 평가 전략

- 소프트웨어 테스트의 단계별 전략과 동일한 개념.
- LLM 메트릭은 API 호출 비용이 들기 때문에 상황에 따라 범위를 조절한다.

| 레벨 | 용도 | 샘플 수 | 메트릭 |
|---|---|---|---|
| L1 | 빠른 검증 (개발 중 수시 실행) | 5건 | hallucination, answer_relevance |
| L2 | 전체 평가 (배포 전 실행) | 15건 | hallucination, answer_relevance, task_completion |

### Hallucination 메트릭 스케일

Opik `Hallucination` 메트릭은 직관과 반대 방향이다.

```
0.0 = 환각 없음 (좋은 점수)
1.0 = 환각 감지 (나쁜 점수)
```

평가 결과에서 0.0이 나오면 "최고 점수"로 해석해야 한다.

### GEval 동작 방식

- `GEval.score(self, output: str, **ignored_kwargs)` — `output` 하나만 받는다.
- `input`, `expected_output` 등 추가 인자는 내부적으로 무시된다.
- **페이로드 패킹 방식**: `input`, `expected_output`, `output`을 하나의 문자열로 묶어 `output`에 전달하면 GEval LLM 판사가 모두 읽고 평가한다.

```python
payload = f"QUESTION: {input}\nEXPECTED_OUTPUT: {expected_output}\nOUTPUT: {output}"
self._inner.score(output=payload)
```

### `expected_output`의 중요성

- `expected_output` 없이 메트릭만 바꾸는 건 효과가 제한적이다.
- 범위 밖 질문(배당, 전망 등)도 데이터셋에 포함해야 한다 — 에이전트가 거절을 잘 하는지도 평가 대상이다.
- `expected_output`은 정확한 값이 아니라 **기대하는 답변 유형에 대한 설명**으로 작성한다.

```json
{"input": "MSFT 배당 정보 알려줘", "expected_output": "배당 정보는 제공하지 않는다는 안내와 함께 제공 가능한 기능을 안내하는 답변"}
```

### `BaseMetric` 상속 패턴

```python
class StockHallucination(BaseMetric):
    def __init__(self, name: str = "stock_hallucination"):
        self.name = name
        self._inner = Hallucination()

    def score(self, output: str, **kwargs) -> score_result.ScoreResult:
        input = kwargs.get("input", "")
        return self._inner.score(input=input, output=output)
```

### 비동기 에이전트를 동기 평가에서 사용하는 방법

- `evaluate()`는 동기 함수만 지원한다.
- 에이전트는 `async`이므로 `asyncio.run()`으로 감싸서 동기 컨텍스트에서 호출한다.

```python
@track
def run_stock_agent(question: str, thread_id: str) -> str:
    async def _stream() -> str:
        ...  # 에이전트 astream() 실행, ChatResponse tool call 감지
    return asyncio.run(_stream())
```

## 구현

### 디렉토리 구조

```
evaluation/
├── data/
│   └── dataset.json                    # 평가 질문 15건 (JSON 분리, expected_output 포함)
├── metrics/
│   ├── stock_hallucination.py          # Hallucination 래핑
│   ├── stock_answer_relevance.py       # AnswerRelevance 래핑
│   └── stock_task_completion.py        # GEval 기반 커스텀 메트릭 (페이로드 패킹 방식)
└── run_eval.py                         # 평가 실행 진입점
```

### 주요 구현 내용

| 항목 | 내용 |
|---|---|
| 데이터셋 | 지원 기능 12건 + 범위 밖 3건 = 15건, `expected_output` 포함, JSON 파일로 관리 |
| 메트릭 | Opik 내장 메트릭 래핑 (`BaseMetric` 상속), `StockTaskCompletion`은 GEval + 페이로드 패킹 |
| 에이전트 호출 | `asyncio.run()` + `astream()` 스트리밍 → `ChatResponse` tool call 감지 |
| 레벨 설정 | `LEVEL_CONFIG` dict → argparse `--level` 인자로 선택 |
| 데이터셋 관리 | 최초 실행 시 Opik에 업로드, 이후 재사용 (`get_or_create_dataset()`) |

### `prompts.py` 개선

평가 결과를 바탕으로 아래 내용을 수정했다.

| 항목 | 변경 전 | 변경 후 |
|---|---|---|
| `get_stock_history` | 프롬프트에 없음 | 언제 사용할지 명시 추가 |
| 수치 생성 금지 | 없음 | "도구 조회 없이 수치 직접 생성 금지" 추가 |
| 간결성 | 없음 | 묻지 않은 개념 설명·추가 안내 문구 금지 추가 |
| 범위 밖 질문 | 단순 거절 | 거절 + 제공 가능 기능 안내 명시 |

### 실행 방법

```bash

uv run python evaluation/run_eval.py --level L1   # 빠른 검증 (5건)
uv run python evaluation/run_eval.py --level L2   # 전체 평가 (15건)
```

## 트러블슈팅

### 1. `ImportError: cannot import name 'AgentTaskCompletion'`

**원인** 공식 문서와 실제 설치된 SDK의 클래스명 불일치.

**해결**
```python
from opik.evaluation.metrics import AgentTaskCompletionJudge  # Judge 붙여야 함
```

---

### 2. `AnswerRelevance` context 필수 오류

**원인** `AnswerRelevance`는 기본적으로 `context`(참조 문서)를 요구한다.

**해결**
```python
self._inner = AnswerRelevance(require_context=False)
```

---

### 3. GEval이 `expected_output`을 무시하여 0점 반환

**증상** `expected_output`을 전달했음에도 GEval이 "평가 대상 답변이 없다"는 이유로 0점.

**원인** `GEval.score(self, output: str, **ignored_kwargs)` — `input`, `expected_output`은 내부적으로 무시된다.

**해결** `score()` 내부에서 페이로드 패킹 후 GEval에 전달.
```python
payload = f"QUESTION: {input}\nEXPECTED_OUTPUT: {expected_output}\nOUTPUT: {output}"
self._inner.score(output=payload)
```

## task_completion 개선 추이

| 단계 | 변경 내용 | 점수 |
|---|---|---|
| 초기 | `AgentTaskCompletionJudge`, expected_output 없음 | 0.14 (8건 0.0) |
| 1차 | `GEval` 교체 | 0.37 (4건 0.0) |
| 2차 | `expected_output` 추가 | 0.55 (1건 0.0) |
| 3차 | 페이로드 패킹 적용 | **0.61 (0건 0.0)** |

데이터셋 설계(`expected_output` 추가)가 메트릭 교체보다 더 큰 개선 효과를 가져왔다.

## 미해결 / 개선 필요 사항

### 뉴스 요약 정확도 문제

`get_recent_news` 툴이 반환하는 뉴스를 에이전트가 요약하는 과정에서 오류가 발생한다.

- **출처 오표기**: The Motley Fool을 "한 증권사"로 잘못 표현
- **무관 기사 포함**: TSLA 뉴스 조회 시 Lucid(경쟁사) 관련 기사가 포함됨

프롬프트 개선만으로는 해결하기 어렵고, 뉴스 데이터 자체의 필터링 또는 에이전트의 뉴스 처리 방식 개선이 필요하다.
