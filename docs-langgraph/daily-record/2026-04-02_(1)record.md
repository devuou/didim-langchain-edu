## 학습

### create_agent / ToolStrategy는 표준 LangChain API가 아님

```python
# stock_agent.py
from langchain.agents import create_agent                        # DeepAgents 라이브러리
from langchain.agents.structured_output import ToolStrategy      # DeepAgents 라이브러리
```

공식 LangChain 패키지에 이 심볼들은 존재하지 않는다. 표준 LangGraph 방식:

```python
from langgraph.prebuilt import create_react_agent
agent = create_react_agent(model, tools, prompt=system_prompt, checkpointer=checkpointer)
```

현재 코드는 강의 제공 DeepAgents 라이브러리 기반이므로, 공식 문서에서 찾으려 하면 찾을 수 없다는 점을 인지해야 한다.

### TYPE_CHECKING 블록의 올바른 역할

```python
# _rag_common.py
if TYPE_CHECKING:
    from elasticsearch import Elasticsearch  # 런타임에 실행되지 않음
```

`TYPE_CHECKING` 블록은 **런타임 import를 막는 것이 아니라**, 타입 체커(mypy/pyright)를 위한 전방 참조 import다. 실제 런타임 import 지연은 각 함수 body 안에서 직접 import하는 방식으로 구현되어 있다.

- `TYPE_CHECKING` 블록 → 타입 힌트 문자열 어노테이션(`"Elasticsearch | None"`)을 타입 체커가 해석할 수 있게 해줌
- 함수 body 내 import → 실제 런타임 lazy init

### 서브 에이전트 기억장치 구조

```
┌─────────────────────────────────────────────┐
│  메인 에이전트 (ReAct)                         │
│  MemorySaver — thread_id별 전체 대화 기억       │  ← 대화 컨텍스트 담당
│                                               │
│  tool call → search_sec_filing(AAPL, 리스크)   │
│                  ↓                            │
│  ┌────────────────────────────────────────┐   │
│  │  서브 에이전트 (StateGraph)              │   │
│  │  checkpointer 없음 → 완전 stateless     │   │  ← 순수 검색 파이프라인
│  │  매 호출마다 새 SecSearchState 생성     │   │
│  └────────────────────────────────────────┘   │
└─────────────────────────────────────────────┘
```

"방금 물어본 종목", "다시 검색해줘" 같은 표현을 에이전트가 올바르게 해석하는 것은 서브 에이전트의 기억이 아니라 **메인 에이전트의 MemorySaver** 덕분이다. 서브 에이전트는 항상 메인 에이전트가 결정한 명시적 파라미터(ticker, query)만 받아 실행된다.

| 기억 공유 방향 | 현재 상태 | 비고 |
|---|---|---|
| 노드 → 노드 (서브 에이전트 내) | ✅ `SecSearchState`로 동작 | fan-in 패턴 |
| 서브 → 메인 | ✅ 문자열 반환 → ToolMessage | 단방향 |
| 메인 → 서브 | ⚠️ ticker, query만 전달 | 추가 컨텍스트 전달 불가 |
| 서브 cross-invocation | ❌ stateless | "추가로 더" 요청 시 중복 제거 불가 |

**향후 개선 검토 필요**: "추가로 더 없어?" 요청 시, 이미 반환한 chunk ID를 기억해 다음 청크를 반환하는 기능 → 서브 에이전트에 checkpointer 추가 + `seen_ids` 필드 필요.

1. **반환 건수 불일치**: LLM이 도구 결과에서 관련 없다고 판단한 뉴스를 제외하면 1건만 남는 경우 발생. 원인은 yfinance `stock.news`가 티커 정확 매핑이 아닌 섹터/키워드 기반으로 뉴스를 집계하기 때문.

### 문제 1: "EV" 키워드 오탐 (단어 경계 미적용)

```python
# 수정 전 — 단순 in 검색
any(kw.lower() in text for kw in keywords)

# 수정 후 — 단어 경계(\b) 기반 regex 매칭
any(re.search(rf"\b{re.escape(kw)}\b", text, re.IGNORECASE) for kw in keywords)
```

`"EV" in "seven"` → `True` (오탐), `"EV" in "ever"` → `True` (오탐) 발생.  
단어 경계(`\b`)를 적용해 정확히 "EV"라는 단어 단위로만 매칭하도록 수정.  
`re.escape(kw)`는 특수문자 포함 키워드("Tim Cook" 등) 안전 처리를 위해 추가.

### Finnhub API 대체 가능성 검토 및 결론

TSLA 뉴스 조회 시 Anthropic 관련 기사가 혼입되는 문제 해결을 위해 Finnhub Company News API 대체 가능성을 평가.

**판단 기준**:
1. 티커 직접 지정 가능 여부
2. 날짜 범위 필터링 지원 여부
3. 뉴스 매핑 정확도 (섹터 기반 vs 티커 직접 매핑)
4. 응답 구조 (필터링 친화도)
5. 도입 비용 대비 기대 효과

**Finnhub vs yfinance 응답 구조 비교**:

| 항목 | yfinance | Finnhub |
|------|----------|---------|
| 호출 구조 | `yf.Ticker(symbol).news` | `GET /api/v1/company-news?symbol=&from=&to=` |
| 날짜 필드 | `content.pubDate` (ISO) | `datetime` (UNIX timestamp) |
| 티커 매핑 | 섹터/키워드 기반 (부정확) | 자체 NLP 엔티티 태깅 (`related` 필드) |
| 필터 방식 | 날짜 범위 파라미터 없음 | `from` / `to` 날짜 파라미터 직접 지정 가능 |
| 응답 구조 | 중첩 객체 (`content{}` 내부) | 플랫 구조 (단순) |

**결론**: Finnhub가 티커 정확도·날짜 필터·응답 구조 면에서 더 우수하나, **도구 교체 보류**.  
이유: `_is_recent`와 `\b` 수정으로 현재 yfinance 구현이 충분히 안정화됨.  
Finnhub 도입 시 API 키 관리, 호출 방식 변경, 테스트 추가 등 비용 대비 추가 효과가 제한적.

## 서브 에이전트 cross-invocation 중복 제거 (seen_ids)

### 문제

`search_sec_filing`은 매 호출마다 새 `SecSearchState`를 생성하는 stateless 구조다. "추가로 더 없어?" 요청 시 이전에 반환한 청크를 모르기 때문에 동일한 청크가 중복 반환됐다.

### 설계

```
thread_id + ticker → cache_key → _seen_ids_cache[cache_key]
                                       ↓
search_sec_filing 호출 시 seen_ids 읽어서 그래프에 전달
                                       ↓
_rerank_fn: seen_ids 제외 후 리랭킹 → 새 seen_ids 반환
                                       ↓
캐시 업데이트 → 다음 호출에서 사용
```

### 핵심 구현

**`SecSearchState`에 `seen_ids` 필드 추가:**
```python
seen_ids: list[str]  # 이미 반환된 chunk ID 목록 (cross-invocation 중복 제거용)
```

**`_rerank_fn` 필터링 로직:**
```python
unseen = [h for h in hits if h["_id"] not in seen_ids]
candidates = unseen if unseen else hits  # 전량 소진 시 fallback
# ... 리랭킹 후
new_seen_ids = list(seen_ids) + [h["_id"] for h in top_hits]
return {"merged_hits": top_hits, "result": format_hits(top_hits), "seen_ids": new_seen_ids}
```

**`search_sec_filing` 캐시 연동 (`InjectedToolArg`):**
```python
@tool
def search_sec_filing(
    ticker: str,
    query: str,
    config: Annotated[RunnableConfig, InjectedToolArg] = None,  # LLM에 노출 안 됨
) -> str:
    thread_id = (config or {}).get("configurable", {}).get("thread_id")
    cache_key = f"{thread_id}:{ticker_upper}" if thread_id else None
    seen_ids = _seen_ids_cache.get(cache_key, []) if cache_key else []
    result = _sec_search_graph.invoke({"ticker": ticker_upper, "query": query, "seen_ids": seen_ids})
    if cache_key:
        _seen_ids_cache[cache_key] = result.get("seen_ids", seen_ids)
    return result["result"]
```

- `InjectedToolArg`: LangChain이 `config`를 자동 주입, LLM 스키마에는 노출되지 않음
- `thread_id` 없으면 `cache_key = None` → 캐시 완전 비활성화 (테스트 환경 등)
- 캐시 키: `"{thread_id}:{ticker_upper}"` — 종목 간 독립 보장

### 수동 테스트 시나리오

| 시나리오 | 질문 흐름 | 확인 포인트 |
|---------|---------|-----------|
| 기본 조회 | `AAPL 사업 구조 알려줘` | Item 1 섹션 정상 반환, 10-K 연도 명시 |
| 추가 조회 | 위 이어서 `추가로 더 없어?` | 1회와 다른 청크 반환 (핵심 검증) |
| 종목 간 독립 | AAPL → TSLA → AAPL 추가 | TSLA 조회가 AAPL seen_ids에 영향 없음 |
| 새 thread | 별도 thread_id로 `AAPL 사업 구조` | 이전 대화 seen_ids 영향 없이 정상 반환 |

로그의 `step: tools` → `content` 필드를 1회·2회 비교하면 청크 교체 여부 확인 가능.

## 평가 시스템 확장 (SEC 서브 에이전트 추가)

### 설계 배경

기존 평가(`run_eval.py`)는 메인 에이전트만 있을 때 설계되어 SEC 서브 에이전트를 평가하지 못했다. 또한 Hallucination 메트릭이 context 없이 실시간 데이터 수치를 판단해 오탐이 많았다 (03-17 기록 참고).

이번에 두 가지 문제를 동시에 해결하는 레이어 분리 구조로 확장했다.

### 평가 레이어 구조

```
Layer 1: 서브 에이전트 단독 평가 (run_eval_sec.py)
  → sec_dataset.json (10건) 사용
  → _sec_search_graph.invoke() 직접 호출
  → SecRetrievalRelevance 메트릭: 검색된 청크가 질문과 관련 있는가?

Layer 2: 메인 에이전트 엔드투엔드 평가 (run_eval.py 확장)
  → dataset.json (21건, +SEC 6건) 사용
  → 기존과 동일하게 메인 에이전트 astream() 호출
  → SecGroundedness 메트릭 추가: search_sec_filing 결과를 context로 캡처해 근거 충실도 판단
```

### 핵심 설계 포인트

#### context 캡처 방식

기존 `run_stock_agent()`는 최종 답변 문자열만 반환했다. `search_sec_filing` tool output을 `context`로 얻으려면 `tools` 스텝에서 캡처해야 한다.

```python
# run_eval.py — astream 루프에서 search_sec_filing tool output 수집
if step == "tools":
    messages = event.get("messages", [])
    for msg in messages:
        if getattr(msg, "name", None) == "search_sec_filing":
            tool_outputs.append(msg.content)

# evaluation_task 반환값에 context 추가
return {
    "input": question,
    "output": output,
    "expected_output": ...,
    "context": tool_outputs,   # SecGroundedness가 이 값을 사용
}
```

`run_stock_agent` 반환 타입을 `str` → `tuple[str, list[str]]`로 변경한 것이 핵심.

#### SecGroundedness: context 없으면 skip

`search_sec_filing`을 호출하지 않은 항목(실시간 주가, 뉴스 등)은 context가 비어 있다. 이 경우 `None`을 반환해 Opik이 해당 항목을 채점에서 제외하도록 설계.

```python
def score(self, output: str, **kwargs) -> ScoreResult | None:
    context = kwargs.get("context", [])
    if not context:
        return None   # search_sec_filing 미호출 항목 → skip
    return self._inner.score(input=..., output=output, context=context)
```

이렇게 하면 Hallucination 메트릭이 "context 없는 실시간 데이터"를 오탐하는 기존 한계를 SEC 항목만큼은 해결한다.

#### SecRetrievalRelevance: 지원 종목 외 케이스 처리

AMZN, GOOGL 등 미지원 종목 질문 시 서브 에이전트는 `"관련 공시 정보를 찾을 수 없습니다."` 를 반환한다. GEval 판단 기준에 이 케이스를 명시 — `expected_output`에 "찾을 수 없습니다"가 포함된 경우, 동일한 안내를 반환하면 높은 점수를 주도록 설계.
