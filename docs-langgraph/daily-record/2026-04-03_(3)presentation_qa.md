### Q1. tiktoken은 LLM을 사용하나요?

아니요. tiktoken은 OpenAI가 공개한 **토크나이저 라이브러리**로, LLM 호출 없이 로컬에서 동작합니다.

텍스트를 토큰 ID 배열로 변환(`encode`)하거나 다시 텍스트로 복원(`decode`)하는 순수 연산이며, 네트워크 요청이나 API 비용이 발생하지 않습니다. GPT-4o 등 OpenAI 모델이 내부적으로 사용하는 것과 동일한 `cl100k_base` 인코더를 사용하기 때문에 "512 토큰으로 청크"하면 실제 LLM이 받는 크기와 정확히 일치합니다.

```python
import tiktoken
enc = tiktoken.get_encoding("cl100k_base")
tokens = enc.encode("AAPL의 주요 리스크는...")  # LLM 호출 없음
chunk = enc.decode(tokens[:512])               # 순수 로컬 연산
```

### Q2. 두 가지 검색 방식을 사용했는데, 질문 예시로 각각 어떻게 검색되는지 설명해줄 수 있나요?

**예시 질문**: `"NVDA의 Blackwell 아키텍처 관련 리스크가 뭐야?"`

**BM25 (키워드 검색)**
- "Blackwell"이라는 단어가 텍스트에 얼마나 자주 등장하는지 빈도 기반으로 점수 산출
- "Blackwell"을 직접 언급한 청크를 높게 평가 → 신제품 출시 리스크, 공급망 관련 섹션 상위 노출
- 장점: 고유명사·신조어에 강함 / 단점: "차세대 GPU 아키텍처"처럼 Blackwell을 직접 언급하지 않은 청크는 못 찾음

**kNN (벡터 검색)**
- 질문 전체를 1536차원 벡터로 변환 후, 의미적으로 가까운 청크를 탐색
- "Blackwell"이 없더라도 "새로운 컴퓨팅 플랫폼 전환 리스크", "GPU 세대 교체 시 수요 불확실성" 같은 청크를 유사 의미로 포착
- 장점: 동의어·패러프레이징에 강함 / 단점: 희귀 고유명사 자체는 벡터 공간에서 뭉뚱그려질 수 있음

→ 두 결과를 병합해 Cohere Rerank로 최종 관련도 재산출 — BM25가 잡은 "Blackwell 직접 언급 청크"와 kNN이 잡은 "의미적으로 관련된 청크"를 함께 평가해 Top 5 선정.

### Q3. 서브 에이전트가 stateless인데, "추가로 더 알려줘" 같은 요청은 어떻게 처리하나요?

서브 에이전트 자체는 매 호출마다 독립 실행되지만, `_seen_ids_cache`(메모리 딕셔너리)가 cross-invocation 상태를 대신 관리합니다.

```
1회 호출: Top 5 청크 반환 → chunk_001~005를 cache에 저장
2회 호출: cache에서 001~005 읽어 seen_ids로 전달 → rerank 단계에서 제외 → 새로운 청크 반환
```

캐시 키는 `{thread_id}:{ticker}` 형태로 종목 간 독립을 보장합니다. 체크포인터 없이 이 패턴만으로 충분한 이유는, 서브 에이전트가 필요한 상태가 "어떤 청크를 이미 봤는가" 하나뿐이기 때문입니다.

### Q4. LLM-as-a-Judge 방식은 신뢰할 수 있나요?

완벽하지 않습니다. judge LLM도 동일한 출력에 대해 매번 다른 점수를 줄 수 있고, judge 자체가 환각을 일으켜 잘못된 평가를 내릴 수 있습니다.

그래서 LLM-as-a-Judge 점수는 **절대적 기준이 아니라 상대적 비교**에 활용하는 것이 적합합니다. 이번 프로젝트에서도 "0.61이 좋은 점수인가"를 판단하기보다, 프롬프트 개선 전후 점수가 올랐는지를 보는 방식으로 사용했습니다.

### Q5. search_sec_filing 결과가 오래된 공시 데이터일 수 있지 않나요?

맞습니다. `ingest_10k.py`를 마지막으로 실행한 시점의 10-K 데이터만 ES에 있으며, 스크립트를 재실행하지 않으면 최신 연도 보고서가 반영되지 않습니다.

현재는 이 한계를 프롬프트로 완화하고 있습니다 — `search_sec_filing` 결과에 `fiscal_year` 필드가 포함되면 에이전트가 답변에 "20XX년 연간 보고서(10-K) 기준"이라고 명시하도록 규칙을 추가했습니다. 근본 해결은 정기적으로 파이프라인을 재실행하거나 EDGAR 신규 제출을 감지해 자동 업데이트하는 구조가 필요합니다.

### Q6. yfinance 키워드 필터에서 등록되지 않은 종목은 어떻게 처리하나요?

`_TICKER_KEYWORDS`에 등록되지 않은 종목(예: AMZN)은 티커 심볼 자체(`"AMZN"`)만 키워드로 사용합니다. 금융 기사에서 `"Amazon (AMZN)"` 형태로 티커가 자주 등장하므로 최소한의 필터 역할은 합니다. 다만 AAPL·TSLA 등 등록 종목보다 필터 정밀도는 낮습니다.

### Q7. 메인 에이전트와 서브 에이전트가 동시에 여러 종목을 처리할 수 있나요?

네. `parallel_tool_calls=True`가 설정되어 있어 "AAPL과 TSLA 리스크 비교"처럼 두 종목을 동시에 물어보면 `search_sec_filing`이 두 번 병렬 호출됩니다. 각 호출은 독립된 `SecSearchState`로 실행되고, `_seen_ids_cache` 키도 `{thread_id}:AAPL`과 `{thread_id}:TSLA`로 분리되어 서로 간섭하지 않습니다.
