## 개요

- LangGraph `StateGraph`를 직접 설계하여 SEC 10-K 공시 문서 검색 서브 에이전트 구현.

## 학습

### StateGraph 직접 설계 vs create_agent()

이전까지는 `create_agent()`가 내부적으로 ReAct 루프를 구성해줬다.
이번에는 StateGraph의 노드·엣지를 직접 정의하는 첫 번째 구현.

| 항목 | create_agent() | StateGraph 직접 설계 |
|---|---|---|
| 흐름 제어 | LLM이 다음 도구를 동적으로 선택 | 개발자가 노드·엣지를 정적으로 정의 |
| 병렬 실행 | 불가 | fan-out/fan-in으로 가능 |
| 사용 적합 | 범용 추론, 다단계 질의응답 | 고정된 파이프라인, 검색·처리 흐름 |

### 병렬 fan-out / fan-in 패턴

```python
builder.add_edge(START, "bm25_search")   # fan-out: 두 노드 동시 실행
builder.add_edge(START, "vector_search")

builder.add_edge("bm25_search", "merge_results")   # fan-in: 둘 다 완료 후 진입
builder.add_edge("vector_search", "merge_results")
```

- `START`에서 동일한 두 노드로 엣지를 추가하면 LangGraph가 자동으로 병렬 실행.
- 두 노드가 모두 완료되어야 `merge_results`가 실행됨 (fan-in).
- 각 노드는 `state`에서 필요한 값을 읽고, 반환하는 `dict`의 key만 State에 업데이트됨.

### subagent-as-tool 패턴

```python
_sec_search_graph = _build_graph()  # 모듈 로드 시 그래프 1회 컴파일

@tool
def search_sec_filing(ticker: str, query: str) -> str:
    """..."""
    result = _sec_search_graph.invoke({"ticker": ticker.upper(), "query": query})
    return result["result"]
```

- StateGraph 전체를 `@tool` 함수로 감싸면 메인 에이전트 입장에서는 일반 도구와 동일.
- 메인 에이전트의 `tools=[..., search_sec_filing]`에 등록하는 것으로 통합 완료.

### tiktoken 기반 토큰 청킹 vs 문자 기반 청킹

| | 문자 기반 (rag_pipeline 샘플) | 토큰 기반 (이번 구현) |
|---|---|---|
| 단위 | 문자 수 (500자) | 토큰 수 (512 tokens) |
| LLM과의 정합성 | 간접적 | 직접적 (cl100k_base 인코더 사용) |
| 구현 | `RecursiveCharacterTextSplitter` | `tiktoken.encode()` + `decode()` |

LLM context window는 토큰 단위로 제한되기 때문에, 청크 크기를 토큰으로 직접 제어하는 방식이 더 정확하다.

### StateGraph 구조 (`sec_search_agent.py`)

```
SecSearchState
  query: str        # 검색 쿼리
  ticker: str       # 대상 종목
  bm25_hits: list   # BM25 결과
  vector_hits: list # kNN 결과
  merged_hits: list # 병합 + 중복 제거
  result: str       # 최종 포맷팅 문자열

START
  ├──→ bm25_search     (ES match 쿼리, ticker 필터)
  └──→ vector_search   (ES kNN, 임베딩 생성 후 검색)
            ↓ (fan-in)
       merge_results   (_id 기준 중복 제거, 높은 score 우선)
            ↓
          rerank       (CrossEncoder 없으면 score 내림차순 fallback)
            ↓
           END
```

### 청킹 방식 비교 및 결정

| | 고정 길이 청킹 (tiktoken) | RecursiveCharacterTextSplitter |
|---|---|---|
| 단위 | 토큰 수 (512 tokens) | 문자 수 (500자 등) |
| 분할 기준 | 토큰 경계 | `\n\n` → `\n` → ` ` 순서로 자연 경계 우선 |
| 문장 보존 | 문장 중간에서 잘릴 수 있음 | 문단/문장 경계를 최대한 유지 |
| LLM 정합성 | 직접적 (context window = 토큰 단위) | 간접적 (문자 수 ≠ 토큰 수) |
| 구현 복잡도 | 낮음 (`tiktoken.encode/decode`) | 낮음 (LangChain 내장) |

**결정: tiktoken 고정 길이 청킹 유지**

10-K 문서는 문단 구조가 불규칙하고 긴 단락이 많아 `RecursiveCharacterTextSplitter`가 더 자연스러울 수 있으나, 이번 구현에서는 아래 이유로 tiktoken 방식을 그대로 사용했다.

- LLM context window는 토큰 단위로 제한되므로, 청크 크기를 토큰으로 직접 제어하는 것이 더 예측 가능하다.
- 최종 LLM에 전달되는 청크 수가 `_FINAL_TOP_N = 5`로 제한되므로, 청크 경계 품질의 영향이 제한적이다.
- 10-K 본문은 이미 섹션별로 분리된 상태이므로 문단 경계 보존의 효과가 일반 문서보다 낮다.

### 데이터 파이프라인 (`scripts/ingest_10k.py`)

```
sec-edgar-downloader → primary-document.html
  → BeautifulSoup.get_text()
  → extract_sections() → {item1, item1a, item7}
  → chunk_text() (tiktoken, 512 tokens, overlap 50)
  → embed_texts() (OpenAI text-embedding-3-small, batch 100)
  → ES bulk upsert (_id = chunk_id → 멱등성 보장)
```

임베딩 재시도 로직 (rag_pipeline 샘플 참고):
- `RateLimitError`: 최대 3회, 지수 백오프 (5 → 10 → 15초)
- `APITimeoutError / APIError`: 동일 처리

## 트러블슈팅

### 1. `download_details=False`로 HTML 파일 없음

**증상**
```
FileNotFoundError: AAPL: .htm 파일을 찾을 수 없음
```

**원인**
`sec-edgar-downloader`의 `download_details=False` 옵션은 `full-submission.txt`만 다운로드하고 개별 HTML 파일을 생성하지 않음.

**해결**
```python
dl.get("10-K", ticker, limit=1, download_details=True)
```
→ `primary-document.html` 생성됨.

### 2. 섹션 추출이 목차(TOC) 텍스트에서 조기 종료

**증상**
`item1`: 1자, `item1a`: 65자, `item7`: 88자 (실제 본문 아닌 목차 항목만 추출)

**원인**
SEC 10-K 문서 구조:
```
[앞부분] 목차(TOC):
  Item 1. Business ............. 1   ← 첫 번째 매칭
  Item 1A. Risk Factors ........ 5

[뒷부분] 본문:
  Item 1. Business             ← 두 번째 매칭 (실제 본문)
  Company Background
  The Company designs...
```
`re.search()`가 첫 번째 매칭(목차)에서 멈춰버림.

**해결**
`re.findall()`로 모든 매칭을 찾고 **마지막 매칭**을 본문으로 사용:
```python
matches = list(re.finditer(start_pat, text, re.IGNORECASE))
m_start = matches[-1]  # 마지막 = 실제 본문
```

### 3. 교차 참조로 인한 섹션 조기 종료

**증상**
`item7` 추출 결과가 227자 (본문 첫 문장에서 잘림)

**원인**
본문 첫 단락에 `"Part II, Item 8 of this Form 10-K"` 라는 교차 참조가 포함되어 있어, stop 패턴 `r"Item\s+(?:7A|8)\b"`이 실제 섹션 헤더가 아닌 교차 참조 텍스트에 매칭됨.

**원인 파악 과정**
```python
repr(text[body_start:body_start+400])
# → '...read in conjunction with...Part II, Item 8 of this Form 10-K...'
```

**해결**
stop 패턴에 `\.` (마침표)를 추가 — 실제 섹션 헤더는 `Item 8.` 형식이고 교차 참조는 마침표 없이 `Item 8`로만 쓰임:
```python
# 변경 전
_STOP_PATTERN = r"Item\s+(?:7A|8)\b"

# 변경 후 (섹션별 개별 지정 + 마침표 포함)
_SECTION_CONFIG = {
    "item1":  (r"Item\s+1\.\s+Business",        r"Item\s+1A\."),
    "item1a": (r"Item\s+1A\.\s+Risk\s+Factors", r"Item\s+(?:1B|2)\."),
    "item7":  (r"Item\s+7\.\s+",                r"Item\s+(?:7A|8)\."),
}
```
→ item7: 18,099자 / 9개 청크로 정상 추출.
